# SDD competitive pulse check — July 2026

**Date:** 2026-07-19
**Scope:** LSA (this marketplace's `lsa` + `core` plugins) vs. the modern spec-driven-development (SDD) landscape.
**Method:** Three parallel research passes (web search + primary-source fetch: GitHub repos/issues/releases, official docs/blogs) against (a) GitHub Spec Kit, OpenSpec, AWS Kiro, Tessl; (b) BMAD-METHOD, Superpowers, Flow-Next, Cline, Roo Code, Aider; (c) vendor-native posture in Cursor, Copilot, Devin, Windsurf, Amazon Q, Claude Code Plan Mode, Codex CLI, plus 2026 SDD discourse (Sean Grove's "The New Code," the "waterfall in Markdown" debate, RTM/IEEE 830 lineage). Every external claim below carries its source; internal claims cite this repo's own files. This supersedes and extends `.lsa/VISION.md` §6 (last updated May 2026 — two months stale against a fast-moving field).

---

## TL;DR

- **The gap LSA was built to close is now a *named, contested* gap in the field** — not a fringe idea. GitHub's own Spec Kit maintainers were asked twice to build a blocking post-implementation verification/reconcile command and declined both times, redirecting to third-party extensions instead. OpenSpec's equivalent (`/opsx:verify`) shipped a July 2026 update and is *still* non-blocking with no hunk-level trace. LSA's `reconcile` is the only surveyed tool that is simultaneously **blocking**, **hunk-level**, and **drift-absorbing** by design, not as an add-on.
- **Kiro (AWS) is the closest vendor-native competitor**, not Spec Kit or OpenSpec — it shares EARS notation and requirement-ID-tagged tasks, but its "spec check" is on-request, not gated, and its traceability is task-level, not hunk-level.
- **Flow-Next (community, MIT, `gmickel/flow-next`) is the closest architectural peer** among tool-agnostic frameworks — a requirement-ID → task → commit → PR coverage table, gating cross-model review, and explicit re-anchoring against drift. It validates LSA's design rather than beating it outright; see § Converged.
- **The honest gaps are mostly self-known and already on the roadmap**, not surprises from this research: the library-spec cache (Tessl's actual product) is scoped and backlogged at `Could` priority; the three tracked metrics have no session-level telemetry yet; "tool-agnostic" is a claim in the spec format, not yet proven in practice (`cursor-equal-support` is still backlog). This research surfaces two *new* gaps worth naming: no cross-model adversarial review (Flow-Next has this), and no explicit RTM/IEEE 830 lineage citation (the field is quietly re-deriving a 40-year-old discipline and LSA doesn't say so).

## 1. Landscape map

Four categories, by what the tool actually *does*:

| Category | Tools | Writes code itself? |
|---|---|---|
| **Spec-orchestration layer** (delegates code-writing) | **LSA**, GitHub Spec Kit, OpenSpec, BMAD-METHOD, Flow-Next | No |

## 3. Where we win — keep and defend

**3.1 The blocking, hunk-traced reconcile gate.** This is now demonstrably not a solved problem elsewhere. Spec Kit's maintainers closed both feature requests for it by pointing to third-party extensions rather than shipping a native command (#1063 closed "as completed" via a community PR; #1745 still open, same redirect). OpenSpec shipped a July 2026 release (v1.6.0) that added a new `/opsx:update` command but left `/opsx:verify` itself unchanged — still, per OpenSpec's own docs, *"a quality gate that runs validation checks without blocking progress."* Kiro's is on-request and task-granular. Only Flow-Next matches the rigor — see §4, this is convergent validation, not a loss.

**3.2 Fact-grounding as a platform-wide, always-on rule — not just an anti-hallucination patch on code.** Tessl's Registry attacks *API* hallucination specifically (library specs). Spec Kit's `/analyze` attacks *internal spec* inconsistency. Nobody surveyed enforces `Statement + Source + searchable quote` on every claim the system makes, in every output, not just code (`.lsa/VISION.md:37`, `core/skills/ground-rules/SKILL.md`). This is a broader claim than any competitor makes about itself.

**3.3 The reconcile-absorbs-drift philosophy is named and load-bearing, not incidental.** OpenSpec's `/opsx:update` and Kiro's sync hooks do something adjacent, but LSA names this explicitly as **Level 2.5** on a rigor ladder — a deliberate, documented choice against both Level 3 (forbid hand-edits) and Level 1 (no reconciliation at all) — with a worked example in `.lsa/VISION.md:145-157`. No competitor documentation surveyed frames drift-handling as a chosen point on an explicit rigor spectrum; it's usually just "a feature."

**3.4 "Deterministic work is scripted" — genuinely not found anywhere else in this survey.** `.lsa/VISION.md` §2 principle 10: any deterministic step (enumeration, set-difference, lookup, tally, format transform) runs as a script the model *cites*, never recomputes at inference time — applied concretely in `reconcile`'s coverage-skeleton script and `verify`'s `resolve-refs.sh` (`lsa/CHANGELOG.md` 0.27.0, 0.28.0). None of the ten competitors researched documents an equivalent principle; this is closer to a compiler-design discipline than an agent-prompting one, and it's a real, defensible differentiator on both cost and reliability grounds.

**3.5 Zero vendor lock, zero external service, runs on the cheapest tier.** Tessl is closed-beta, VC-backed, business model undisclosed. Kiro is AWS-proprietary — *"the tradeoff is vendor lock: you adopt their IDE, their model pipeline, their ecosystem"* ([Augment Code](https://www.augmentcode.com/tools/best-kiro-alternatives)). LSA is five Markdown plugins plus one read-only shell hook, `rm -rf .lsa/` to fully detach (`lsa/README.md:163`), and runs the full loop on Claude Pro at Sonnet with nothing gated behind a paid tier (`README.md:134-143`).

**3.6 The core/pack inversion is a genuinely distinct positioning claim.** BMAD-METHOD is confirmed still actively developed at v6.10.0 (released 2026-07-03, 50,808 GitHub stars — verified directly via `gh repo view bmad-code-org/BMAD-METHOD`) and its public identity remains a software-development core with domain "expansion packs" layered on top — the pattern `.lsa/VISION.md:75` names and inverts. (The specific pack names an earlier research pass attributed to a July-2026 doc page could not be independently re-confirmed — that page 404'd on refetch — so this claim rests on BMAD's well-established core+expansion-pack architecture generally, not those specific pack names.) LSA inverts the pattern deliberately: the domain-neutral discipline is the core, and *even software development is a pack* (`.lsa/VISION.md:75-77`). No competitor surveyed makes this move.

## 4. Converged — good company, sharpen the edge

- **Workflow shape.** Superpowers' brainstorm → plan → subagent-per-task → TDD red-green-refactor → review independently arrived at nearly LSA's shape, and it's now in Anthropic's official plugin marketplace. Its star count is genuinely startling — 257,451 stars / 22,929 forks, independently confirmed via `gh repo view obra/superpowers` (GitHub's own API, not a secondhand claim) — putting a single-maintainer Claude Code plugin ahead of nearly every open-source project on GitHub by that metric. The number itself checks out; what it *means* is a separate question this research can't settle (organic adoption vs. a star-campaign effect are both well-documented phenomena on GitHub for repos that go viral in AI-tooling circles). Convergent evolution across two unrelated teams is still a good sign for the workflow shape; it is not a differentiator LSA should lead with, and the star count should not be read as a market-validation signal either way.
- **Re-anchoring / drift discipline as an industry instinct.** Flow-Next's re-anchoring explicitly cites *"Anthropic's long-running agent guidance"* that agents must re-read sources of truth because "reads are cheap while drift is expensive" — the same instinct as LSA's SessionStart drift hook and reconcile's absorb-not-forbid model. Kiro's sync hooks and OpenSpec's `trace`/`/opsx:update` are the same instinct at lower rigor.
- **EARS notation.** Kiro's `requirements.md` uses full-file EARS; LSA converged on the same standard independently (`.lsa/VISION.md` §6 item 1, RESOLVED May 2026) but scoped it narrower — only the acceptance-criteria block, not the whole requirement narrative. Worth naming honestly: Kiro's adoption is broader than ours.
- **The "waterfall in Markdown" critique has a documented rebuttal that matches LSA's own design, unremarked.** The dominant 2026 backlash (François Zaninotto, *"Spec-Driven Development: The Waterfall Strikes Back"*, [marmelab.com](https://marmelab.com/blog/2025/11/12/spec-driven-development-waterfall-strikes-back.html)) is answered by Marc Brooker (*"Spec Driven Development isn't Waterfall"*) and Roger Wong on the same point: the discriminator is whether specs stay *mutable* through implementation. LSA's reconcile-absorbs-drift design is a direct, working answer to this critique — but `.lsa/VISION.md` never engages the critique by name. Free credibility left on the table (see §6).

## 5. What we lack

**5.1 Already self-known, already roadmapped (not news, but worth restating honestly):**

- **The library-spec cache** — Tessl's actual core product (10,000+ pre-built, version-pinned library specs, fetched before generation, zero regeneration cost) is exactly what `.lsa/VISION.md:214-230` scoped down to "3–5 most-used libraries, pinned once" and explicitly declined to build at Tessl's scale. Still `status: backlog`, `priority: Could` (`.lsa/roadmap.yaml:21-27`). This is the one item where a direct competitor has already shipped, at scale, the exact idea we deliberately shrank.
- **The three tracked metrics stopped being measured — this is a regression, not an aspiration.** `.lsa/VISION.md §5` commits to tracking accuracy-to-task, citation density, and scope-creep rate, and `core/CHANGELOG.md` 0.20.0 states plainly: *"Load rate is not instrumented — no session-level telemetry exists."*

  **Correction (added 2026-07-19 during pitch shaping — this section's first draft called the metrics "aspirational, not measured," which is wrong):** the layer *was* built and did run. `.lsa/metrics.md` exists today, carrying the correct three-metric schema, an explicit no-statistics note (*"Pass/fail counts only — no statistical eval"*), and the line *"Written by `lsa:verify` on clean PASS"* — with **exactly 2 rows, both dated 2026-05-21**. It stopped because `lsa` 0.16.0 (2026-06-08) removed the writer as refactor collateral: *"**`verify`** — refocused to the *before*-delegation grounding check … **Removed the orphan-AC / `metrics.md` machinery**"* (`lsa/CHANGELOG.md`), and the second writer lived in the `sync` skill, which the same release deleted (`lsa/skills/` now holds seven skills, no `sync`).

  Since then **23 `conformance.md` files** have accumulated, each containing an orphan-hunk list — the per-cycle scope-creep signal — computed by `reconcile` and then discarded. That inverts §6 item 4's cost estimate below: the metrics are not expensive-because-unbuilt, they are cheap-because-the-raw-material-is-already-being-produced-and-thrown-away. It also means `.lsa/metrics.md` is currently *misleading on its own* — a reader would reasonably infer those two May features are the only Extended-flow cycles ever archived. Shaped as `restore-tracked-metrics-harvest`.
- **"Tool-agnostic" is a claim about the spec format, not a proven property.** `cursor-equal-support` remains `status: backlog` (`.lsa/roadmap.yaml:560-566`) — in practice, every LSA cycle to date has run inside Claude Code. Spec Kit supports 30+ agents natively today; Flow-Next supports five platforms including a community OpenCode port. LSA's tool-agnosticism is real at the *format* layer (EARS + Gherkin are genuinely portable) but unproven at the *distribution* layer.
- **Cost/pacing friction on Pro tier is real and unresolved.** `reduce-sub-agent-dispatch-fan-out-for-sonnet` is `in_progress` (`.lsa/roadmap.yaml:444-452`); the README's own caveat — *"a full Extended LSA cycle or a parallel `manager:implement` run is token-heavy"* (`README.md:143`) — is an honest admission competitors with hosted infrastructure (Kiro, Tessl, Devin) don't have to make, because their compute cost is abstracted into a subscription rather than felt turn-by-turn in the user's own context budget.
- **Statistical eval rigor deliberately deferred.** `.lsa/VISION.md:232-246`: "genuinely overkill for v1, defer" — no Wilson-CI or Elo-style variance tracking. Community tooling has this; LSA doesn't, by choice, not oversight.

**5.2 Newly surfaced by this research (not previously named in VISION.md):**

- **No cross-model adversarial review — and the obvious fix is blocked by our own safety floor.** Flow-Next's gating review requires a *different* LLM to issue "SHIP" before work proceeds — a real defense against a model rubber-stamping its own output. LSA's `reconcile` is "independent" only in the sense of a fresh context with no write access to the artifacts it grades (`lsa/README.md:100`); nothing records or requires that it run on a different model than `delegate` used.

  **Correction (added 2026-07-19 during pitch shaping, superseding this section's first draft and §6 item 2 as originally written):** the intuitive fix — "route `reconcile` to a different tier than `delegate`" — is *unimplementable* against the shipped contract. `lsa/knowledge/model-routing.md` §"Resolution algorithm" step 1 places the `lsa:reconcile` grader in a **floored set**: it resolves `inherit` and stops, and *"a map entry naming a lower tier for a floored surface is ignored"* — locked by an adversarial scenario (`.lsa/features/pro-tier-token-affordability/model-routing/flow-3-grader-floor.feature`: *"Given .lsa.yaml routing maps lsa:reconcile to haiku # adversarial … Then the resolved tier is inherit, not haiku"*). On Pro there is only one capable tier; on Max the session model is Opus, so forcing reconcile to *differ* means routing the grader **down** to Sonnet — precisely the downgrade the floor exists to forbid.

  The axis that can actually vary is the **implementer**, not the grader: `lsa:delegate` is explicitly external (*"Claude Code, Cursor, Copilot, or a human"*, `lsa/README.md:10`). In any run where the implementer isn't the session model, cross-model grading *already holds* — LSA simply has no field that records it. So the real gap is narrower and cheaper than first stated: not missing enforcement, but an **unrecorded property**. Shaped as `observable-grader-independence`.
- **No explicit RTM / IEEE 830 lineage.** The field is visibly re-deriving 1990s-era requirements-traceability-matrix discipline under new names — Kiro's requirement-ID task tags, the community RTMX project (*"requirements traceability as a CSV file in git... status derived from test results, not manually updated,"* [github.com/rtmx-ai/rtmx](https://github.com/rtmx-ai/rtmx)), academic RTTM surveys. `.lsa/VISION.md` cites Spec Kit, OpenSpec, Kiro, and Tessl as its comparison set but never cites the requirements-traceability lineage it is, in substance, reimplementing. This is a credibility gap, not a functionality gap — LSA's `conformance.md` coverage table *is* an RTM; saying so costs nothing and buys decades of prior-art legitimacy.

  **Precision note (added 2026-07-19 during pitch shaping):** this section's first draft said the discipline is *"codified in IEEE 830"*, which is loose. The accurate split: IEEE 830-1998 and its successor ISO/IEC/IEEE 29148 establish **traceability as a required property** of requirements; they do not specify the RTM table format, which became the conventional instrument for demonstrating that property largely via safety/regulatory regimes (DO-178C, IEC 62304). So *"conformance.md is a requirements traceability matrix"* is true and claimable; *"LSA implements IEEE 830 / ISO 29148"* is **not** — LSA has been audited against neither. Both standards are paywalled and were not read directly here, so the lineage detail is `[unverified]` at clause level and must be cited by name/number/year only. Two properties of `conformance.md` actually *exceed* ordinary RTM practice (status derived from execution rather than manually maintained; hunk-level rather than file/module-level granularity); the honest gaps are no upstream trace to originating stakeholder need, no verification-method taxonomy, no NFR rows, and no requirement attributes. Shaped as `name-conformance-as-rtm`.
- **No adoption or external-validation signal of any kind.** Spec Kit ships near-daily releases backed by GitHub. BMAD-METHOD: 50.8k stars, 38 releases. Superpowers: accepted into Anthropic's official marketplace. Flow-Next: small but real (Discord, sponsorship program, 1,149 commits). LSA has none of this — it is a single-maintainer, personal system with no external users, no issue tracker activity from strangers, no citation from any third party found in this research. That's consistent with its stated scope ("personal-use first," `README.md:132`), but it means every claim in §3 above is currently unverified by anyone outside this repo.
- **No IDE-native experience.** Kiro, Cursor, Windsurf are IDEs with inline UI for plans, diffs, and hooks. LSA is Markdown files plus CLI/plugin slash commands — functionally complete, but the interaction surface is strictly behind any IDE-native competitor's polish, and this is not on the roadmap at all (not even as a "Could").

## 6. Feasibility to push further — concrete, effort-graded

Ranked by (impact on the actual differentiator) ÷ (effort), using only moves grounded in gaps found above:

1. **Cite the RTM/IEEE 830 lineage.** Effort: near-zero (one `.lsa/VISION.md` paragraph + one `lsa/README.md` line). Impact: converts "we invented a coverage table" into "we correctly implemented a 40-year-old V&V practice for the agent era" — the more credible claim, and it costs nothing to make since `conformance.md`'s requirement↔hunk table already *is* an RTM.
2. ~~**Record cross-model status on the reconcile verdict.**~~ **ABANDONED 2026-07-19 — this recommendation was wrong twice, and the second error is the instructive one.**

   *First wording:* "reconcile floors to a different tier than delegate used." Withdrawn — the grader is a floored surface and cannot be routed down (see §5.2).

   *Second wording:* "record it on the implementer axis instead; cross-model grading already happens in many runs, we just don't record it." **Also false.** `lsa:delegate` is in the *same* floored set as `lsa:reconcile` — both "resolve `inherit` and stop" — so an Agent-dispatched implementer **is** the session model. The two can never differ. An external implementer (Cursor/Copilot/human) is unobservable, yielding `unknown`. So **`cross-model: yes` is unreachable**, and the feature would ship a three-valued field where one value can never occur. Routing *up* cannot rescue it: the vocabulary is `inherit | sonnet | haiku` with nothing above `inherit`, and hardcoding `opus` is banned (`.lsa/standards/code.md:52`).

   The pitch (`observable-grader-independence`) was shaped, approved, decomposed, and fully specified — 14 grounded requirements — before `lsa:specify` disproved the premise. Dropped at that gate; the spec is retained as the record of *why* this is architecturally unreachable. **The gap vs Flow-Next stays open**, and the only path to closing it is having `delegate` ask the user to declare an external implementer's model — the pitch's rejected Fork 3b, which needs the pitch reopened.

   **Process note worth keeping:** this error originated in *this file*, survived shaping and decomposition unchallenged, and died at the before-build grounding gate — which is precisely what that gate is for. Had the batch gone straight to a delegate session, a smaller model would have implemented all 14 requirements faithfully and the defect would have surfaced at reconcile or later.
3. **Ship the library-spec cache for 3–5 pinned libraries.** Effort: medium — already scoped in VISION.md, already a backlog row (`.lsa/roadmap.yaml:21-27`), `priority: Could`. Impact: closes the one gap where a funded competitor (Tessl) has already validated the idea at scale; even a shrunk version removes a known "why not just use Tessl" objection.
4. **Instrument the three tracked metrics for real.** Effort: medium-high — needs session-level logging LSA doesn't have today (`core/CHANGELOG.md` 0.20.0 admission). Impact: turns every claim in §3 from narrative into measured; this is the highest-leverage move for external credibility but the most expensive to build.
5. **Promote `cursor-equal-support` off the backlog.** Effort: already epic-decomposed (`.lsa/roadmap.yaml:560-566`: `generate-script`, `vision-second-surface`, `generate-and-copy-docs`). Impact: converts the tool-agnostic *claim* into a tool-agnostic *fact*, directly answering the gap in §5.1.
6. **Publish this comparison, specifically the Spec Kit #1063/#1745 finding, as content.** Effort: near-zero (this document exists). Impact: "the exact command GitHub's own maintainers were asked twice to build and declined to, we built and made blocking" is a genuinely sharp, sourced, differentiated claim — but only useful if LSA ever seeks users beyond one person, which is explicitly not the current goal (`README.md:132` "Personal-use first").

Items 1–3 are cheap and directly reinforce the actual moat (§3.1–3.4). Item 4 is the expensive one and the one that would matter most if this system ever needs to prove itself to anyone but its own maintainer. Item 6 is optionality, not a recommendation — it only matters if the scope commitment in `README.md:132` changes.

## 7. Honest verdict

The core bet — that a blocking, hunk-traced, drift-absorbing verification layer is the load-bearing differentiator, not the spec-authoring pipeline itself — holds up better after this research than before it. It is not a crowded field on that specific dimension: two vendors explicitly declined to build it (Spec Kit), one vendor built a weaker version and hasn't strengthened it in two months of releases (OpenSpec), and only one community project (Flow-Next) matches the rigor, which is validation rather than a threat. Where LSA is genuinely behind is adoption, telemetry, and IDE polish — none of which are architecture problems, all of which are either explicitly out of current scope (adoption — personal-use-first) or already correctly triaged on the backlog (telemetry, cross-tool support). The one clean miss this research found that wasn't already self-known is the missing RTM lineage citation, which is a one-paragraph fix with real credibility upside.
