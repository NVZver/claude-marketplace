# LSA v0.2.0 + Core v0.2.0 — Design

**Date:** 2026-05-20
**Status:** Approved (in design dialog 2026-05-20); pending written-spec review
**Parent vision:** `vision/VISION.md` v0.4
**Parent design:** `vision/specs/archive/2026-05-20-core-v1/design.md` (core v0.1.0 — the spine this release extends)
**Scope:** Bring `lsa` and `core` to Vision-aligned completeness so LSA can actually be run on this repo. Closes the seven gaps between `lsa` v0.1.1 and Vision v0.4 (the four `[Unreleased]` items in `lsa/CHANGELOG.md` plus three Vision-derived gaps: tier model, reconcile loop, self-improvement metrics).

---

## 1. Goal

Ship `core` v0.2.0 and `lsa` v0.2.0 such that:

1. LSA's full workflow (Phase 0 Discover + Phases 1–7 + ad-hoc Reconcile = 8 numbered phases plus 1 ad-hoc) can be exercised against **this repo** (markdown-only, `vision/`-prefixed) without forking the plugin.
2. The reconcile loop from `vision/VISION.md:135` works end-to-end — direct edits to plugin artifacts are detected and proposed for spec absorption rather than blocked.
3. Tier-aware ceremony works as Vision §4 (`vision/VISION.md:116`) describes — T1 single-pass / T2 light loop / T3 full LSA — with the orchestrator's chain-of-thought visible and human-confirmed.

The success test: a session on this repo `(a)` triggers `core/tier-selector` on every non-trivial task, `(b)` runs `lsa-verify` in doc-mode against `core/skills/**/SKILL.md`, `(c)` detects drift introduced by direct skill-file edits and surfaces a `lsa-reconcile` proposal, `(d)` writes a `metrics.md` artifact at verify time, all driven by a `/.lsa.yaml` that points at `vision/VISION.md` and `vision/specs/`.

## 4. The three new skills in detail

### 4.1 `core/skills/tier-selector/SKILL.md`

**Frontmatter (final shape):**

```yaml
---
name: tier-selector
description: Apply before any non-trivial task — when the work touches behavior, adds a new module, changes an API or data model, exceeds ~5 files, or lacks an existing spec. Outputs a tier (T1 / T2 / T3) with visible chain-of-thought reasoning over boundary signals, then waits for human confirmation. Per Vision §4: ceremony scales to the weight of the task.
---
```

The description front-loads trigger nouns (*"non-trivial"*, *"behavior"*, *"new module"*, *"API"*, *"data model"*, *"~5 files"*, *"existing spec"*) so Claude Code's description-match fires early. The 1,536-char combined-with-when_to_use cap noted in `vision/specs/archive/2026-05-20-core-v1/design.md:78` is respected.

**Body (Actor shape, ≤ 90 lines):**

1. **Goal** — Classify the requested task into T1, T2, or T3, present the reasoning, and wait for human confirmation before any ceremony fires.
2. **Input** — A natural-language task description (the user's most recent ask).
3. **Steps** —
   1. List the **boundary signals** present in the task (checklist, verbatim from `vision/VISION.md:124`): new module? API/contract change? data-model change? > ~5 files? spec exists?
   2. Apply the **classification table** (verbatim from `vision/VISION.md:128`): one-string typo → T1; bug in spec'd module → T2; new behavior crossing 3 boundaries → T3; mechanical wide-shallow → T2.
   3. State the chain-of-thought as a one-paragraph summary keyed to the signals from step 1.
   4. Propose tier + summary to the human; **stop**.
   5. On confirm: hand off — T1 returns control to the agent for direct response; T2 invokes `lsa-discover`; T3 invokes `lsa-discover` then `lsa-specify`. **Note (extension to Vision §4):** Vision's T3 loop is *"specify → plan → implement → verify → sync"* (`vision/VISION.md:120`) with no discover step. v0.2.0 extends T3 with `lsa-discover` upfront so light Q&A is universal across T2 and T3, and `lsa-specify`'s own clarification block becomes optional rather than the only entry point. Recorded as a Vision §4 amendment in design §15 follow-ups.
4. **Output** — A tier label (`T1` / `T2` / `T3`) plus a 2–4 sentence rationale. Human-readable. Confirmed before downstream skills fire.
5. **Constraints** —
   - Do not start LSA ceremony before tier confirmation.
   - Do not invent boundary signals not actually present in the task description.
   - Do not silently choose a higher tier than the human picks — log the override and proceed at the human's tier.

**Knowledge-vs-Actor note.** The boundary signals and classification table are arguably Knowledge. Per Decision (§2), they ship embedded inside Steps because "apply these signals" is procedure — the table is read *during* execution, not as a standalone reference. Vision §3 (`vision/VISION.md:54`) is the authority; this trades off purity for one-file simplicity. Revisit if a second skill ends up restating the same signals.

**Footer line:** *"On confirm, downstream LSA skills (`lsa-discover`, `lsa-specify`) absorb the tier into their own gates. Every output still obeys `ground-rules`."*

### 4.2 `core/CLAUDE.md` (always-on tier fragment, ships in `core`)

A small Markdown fragment plugins can merge into the host project's `CLAUDE.md` (per Vision §3 `vision/VISION.md:106` — *"`ground-rules` (always-on) ships as a `CLAUDE.md` fragment"*; tier-selector extends this pattern). Content:

```markdown
## Tier selection (always-on)

Before any non-trivial task, invoke `core/tier-selector` to classify the work as T1, T2, or T3 — and present the reasoning to the human for confirmation. Skip only for tasks that obviously stay inside T1 boundaries (single-string edits, single-question answers).

**The boundary signals (cite Vision §4 `vision/VISION.md:124`):** new module · API/contract change · data-model change · ~5 files · no existing spec.

Tier outcomes:
- **T1** — single pass, no LSA ceremony. `ground-rules` still applies.
- **T2** — `lsa-discover` (light) → agent TDD → `lsa-verify`.
- **T3** — `lsa-discover` → `lsa-specify` → `lsa-plan` → implement → `lsa-verify` → `lsa-sync`.
```

This fragment is installed by the human into their project's `CLAUDE.md` (or read directly from `core/CLAUDE.md` if their host loader supports plugin-level CLAUDE.md merging). For this repo specifically, the slimmed `/CLAUDE.md` (§11) embeds this fragment verbatim.

### 4.3 `lsa/skills/lsa-discover/SKILL.md`

**Frontmatter:**

```yaml
---
name: lsa-discover
description: Light discovery phase used at the start of every T2 and T3 task. Asks the minimal clarifying questions needed to identify the affected module spec, the change's intent, and the acceptance criterion. Writes nothing to disk for T2 (oral output only); hands the captured context to lsa-specify for T3. Use this before any code or spec change when the tier is T2 or T3.
---
```

**Body (Actor shape, ≤ 70 lines):**

1. **Goal** — Establish minimum-viable context: which module, what's the change, what's the AC. Cheap for T2; the front-half of T3's deeper discovery.
2. **Input** — Task description from `tier-selector`'s confirmed handoff; the current `.lsa.yaml` (for module list).
3. **Steps** —
   1. Read `.lsa.yaml` `modules.*.spec` paths; list module names. Observable: the candidate-module list printed back to the human.
   2. Ask the **three-question discovery probe**: (a) Which module(s) does this touch? (b) What's the change in one sentence? (c) What's the acceptance criterion in one sentence? Observable: three short answers captured.
   3. For T2 only — output a single-paragraph context summary; **stop** there. The agent then writes a failing test, implements, and runs `lsa-verify`.
   4. For T3 only — write the answers as a draft `discovery.md` block; hand off to `lsa-specify` which expands them into the full feature spec. Observable: `discovery.md` written under the feature's working directory (a scratch file, not committed).
4. **Output** —
   - T2: short context paragraph (oral).
   - T3: `discovery.md` scratch file, plus a one-line handoff message naming the chosen module(s) and feature name.
5. **Constraints** —
   - Three questions, no more. If the agent needs deeper context, escalate to T3 (re-invoke `tier-selector`).
   - Do not write to `vision/specs/` (or the configured `specs_root`) — that's `lsa-specify`'s job.
   - Do not invent module names not present in `.lsa.yaml`.

### 4.4 `lsa/skills/lsa-reconcile/SKILL.md`

**Frontmatter:**

```yaml
---
name: lsa-reconcile
description: Absorbs direct artifact edits (the human edited a SKILL.md, a config, a plugin file by hand) into the matching module spec, rather than blocking the edit. Compares current artifact_paths against the last-sync commit recorded in .lsa-sync-state.json, summarizes the drift per module, and proposes a one-line spec update for each. Human confirms each absorption individually. Use after direct edits, or when SessionStart warns of drift.
---
```

**Body (Actor shape, ≤ 80 lines):**

1. **Goal** — Close the drift between artifact reality and module specs by absorbing edits into the spec — Level 2.5 (`vision/VISION.md:138`).
2. **Input** —
   - `.lsa.yaml` (module/artifact mapping).
   - `.lsa-sync-state.json` (last-sync commit SHA per module).
   - The current git state.
3. **Steps** —
   1. For each module in `.lsa.yaml`, compute `git diff <recorded-sha> -- <artifact_paths>` (working-tree vs recorded SHA — catches both committed and uncommitted drift; the trailing `..HEAD` form misses uncommitted edits, which is the very case Vision §4's reconcile example surfaces). Observable: a per-module diff summary printed (file count + line count).
   2. If no module has drift, exit with "no drift detected" and **stop**.
   3. For each module with drift, **classify the delta** as either (a) *change to an existing behavior* (the spec already has a requirement that's now contradicted by the artifact) or (b) *new behavior* (the artifact introduces something the spec doesn't cover). Observable: a draft "spec delta" block written to the working scratch, with the classification noted.
   4. Present each delta to the human for **per-module confirm** (`y/n` per module). Hard confirm — no implicit approvals.
   5. On confirm for a module — **reverse-sync per Vision §4** (`vision/VISION.md:143` — *"reverse-sync — the spec absorbs reality"*):
      - Class (a) — *update in place*: edit the contradicted requirement line(s) so the spec now states the new behavior. Replace the old value, don't append a new value next to it. Vision's example (`vision/VISION.md:141`): the spec said *"sessions expire at 30 days"*; on confirm the spec is edited to *"sessions expire at 7 days"*. Tag the edited line with `<!-- reconciled: YYYY-MM-DD -->`.
      - Class (b) — *append new requirement*: add a new requirement in the appropriate section of the module spec, tagged `<!-- reconciled: YYYY-MM-DD -->`. (This is true append, not the "Drift absorbed" heading approach which is rejected — that approach leaves the spec self-contradictory.)
      - Update `.lsa-sync-state.json` with the new HEAD SHA for that module. Observable: spec file edited (specific line(s) shown in the diff); state file updated.
   6. On reject for a module: leave the spec untouched. Optionally write the rejected delta to `research-backlog.md` for later attention.
4. **Output** — Updated module specs (those confirmed, in-place or appended per classification), updated `.lsa-sync-state.json`, summary of what was absorbed and what was rejected.
5. **Constraints** —
   - **Never block, revert, or reformat the artifact edits themselves.** Vision §4 (`vision/VISION.md:144`): *"It does NOT block or revert."*
   - **Never leave the spec self-contradictory.** If an existing requirement contradicts the absorbed change, *replace* the contradictory value (Class a). Do not append a new value while leaving the old one in place.
   - One module at a time — never bundle multiple modules into a single confirm.
   - If `.lsa-sync-state.json` is missing, treat as if the recorded SHA is the first commit on `main` (initialize, don't error).

## 5. Existing LSA skills — refactor scope

Per Decision §2, the skill-shape decision is **mechanical relabel**: existing step content is preserved; section headers are added/renamed to match `core/actor-template`'s Goal / Input / Steps / Output / Constraints shape (current LSA skills use ad-hoc `## Step 1 — X` headers — those collapse under one `## Steps` block with numbered sub-items).

**Two skills exceed mechanical scope** — `lsa-verify` adds a whole new branch (doc-mode), `lsa-sync` adds a new file write (sync-state). These are flagged below as **substantive** rather than mechanical, with explicit acknowledgement that their plan tasks (Task 19 and Task 20) are mid-sized rewrites, not relabels.

**Hardcoded paths across all six skills.** Every existing LSA skill currently hardcodes `/CLAUDE.md`, `/specs/...`, and (for `lsa-verify`) `/src/` in its Step 1 read list and in shell commands. Verified at: `lsa/skills/lsa-specify/SKILL.md:21-23`, `lsa/skills/lsa-verify/SKILL.md:17-31`, `lsa/skills/lsa-sync/SKILL.md:14-20,74`, `lsa/skills/lsa-plan/SKILL.md:15-21`. The refactor **must replace these with reads from `.lsa.yaml`** (`constitution`, `specs_root`, `modules.*.artifact_paths`) — otherwise `.lsa.yaml` does nothing in practice. The plan enumerates the path replacements per skill in Phase B tasks.

| Skill | Refactor class | Changes |
|---|---|---|
| `lsa-init` | Mechanical + path swap | Header relabel; reads `.lsa.yaml` (or applies defaults: `constitution: /CLAUDE.md`, `specs_root: /specs/`, `mode: code`); brownfield mode generalized to scan `modules.*.artifact_paths` instead of hardcoded `/src/`. |
| `lsa-specify` | Mechanical + path swap | Header relabel; Step 1 reads `${constitution}`, `${specs_root}/main.spec.md`, `${specs_root}/modules/<name>/spec.md` (not the hardcoded paths). Accepts optional `discovery.md` from `lsa-discover`. |
| `lsa-plan` | Mechanical + path swap | Header relabel; Step 1 reads use `${constitution}` and `${specs_root}/...`. |
| `lsa-verify` | **Substantive** | Header relabel; Step 1 reads use `${constitution}` and `${specs_root}/...`. **New: doc-mode branch (§8).** When `.lsa.yaml: mode` is `docs` or `mixed`, diff each module's `artifact_paths` against `main` instead of (or in addition to) `git diff main -- src/`. **New: writes `metrics.md` to the feature archive on PASS (§9).** Tasks 19 is a mid-sized rewrite, not a relabel. |
| `lsa-sync` | **Substantive** | Header relabel; Step 1 reads use `${constitution}` and `${specs_root}/...`; Step 5 archive command uses `${specs_root}/archive/...`. **New: writes `.lsa-sync-state.json` (records HEAD SHA per module).** **New: writes `metrics.md` aggregate if `lsa-verify` wrote per-feature rows.** Task 20 is a mid-sized rewrite. |
| `lsa-revise-constitution` | Mechanical + path swap | Header relabel; reads/writes the path configured by `.lsa.yaml: constitution` (default `/CLAUDE.md`), not hardcoded. |

**Marker sweep (gap 4).** Across all 6 reshaped skills plus the 2 new skills, every instance of `[ASSUMPTION: ...]` (uppercase, current LSA convention) becomes `[assumption: <why>]` (lowercase, with inline reason). Every `[INFERRED — verify]` (used in `lsa-init` brownfield mode) becomes `[assumption: inferred from <source>; verify]`. The marker `[cannot verify]` is introduced where uncertainty is total (no inference possible). All three forms align with `core/skills/ground-rules/SKILL.md` per `lsa/CHANGELOG.md:11`.

## 6. `.lsa.yaml` schema

Single config file at the project's repo root. All keys optional except `modules` (which defaults to empty). Defaults apply when a key is absent.

```yaml
# .lsa.yaml — Living Spec Architecture configuration
# Schema version: 1 (matches lsa plugin major version 0.x.y)

constitution: vision/VISION.md
  # Path to the constitution document. Default: /CLAUDE.md
  # Read by every LSA skill before acting.

specs_root: vision/specs/
  # Root directory for the LSA specs tree. Default: /specs/
  # All sub-paths (modules/, features/, standards/, archive/) live under this.

mode: docs
  # docs | code | mixed. Default: code.
  # docs  — no /src/; verification compares artifact_paths to specs only.
  # code  — current LSA behavior; verification compares /src/ to specs.
  # mixed — both; doc-mode applied to artifact_paths, code-mode to /src/.

modules:
  # Map of <module-name>: { spec, artifact_paths }
  # Required for verification to function in doc-mode or mixed mode.

  <module-name>:
    spec: <path-relative-to-repo-root>
      # Path to this module's spec.md (under specs_root/modules/<name>/).

    artifact_paths:
      - <glob-pattern>
      - <glob-pattern>
      # Glob patterns (relative to repo root) for files that implement this module.
      # Verification diffs these against the matching module spec.
```

**Schema-version note.** The top-of-file comment (`# Schema version: 1`) is informational. If a future LSA major (1.x.y) introduces breaking schema changes, that comment becomes a hard `schema_version: N` key for forwards-incompatibility detection. v0.2.0 ships without the key — additions to v1 schema are non-breaking.

**Defaults when `.lsa.yaml` is absent.** LSA falls back to the v0.1.1 behavior: `constitution: /CLAUDE.md`, `specs_root: /specs/`, `mode: code`, `modules: {}` (empty — verification skipped). This means the v0.1.1 behavior is fully preserved for projects that don't opt into `.lsa.yaml`.

**This repo's `/.lsa.yaml` (§11):**

constitution: vision/VISION.md
specs_root: vision/specs/

Changes to marketplace.json are tracked by git but don't trigger LSA verify. Documented in `/specs/main.spec.md` once written.

## 7. The reconcile loop

**Detection (cheap, git-native).** On every `lsa-sync` and `lsa-reconcile`, write the current HEAD SHA to `.lsa-sync-state.json` **at the repo root** (sibling of `.lsa.yaml`):

```json
{
  "modules": {
    "core": { "last_sync_sha": "abc123...", "last_sync_iso": "2026-05-20T15:42:00Z" },
    "lsa":  { "last_sync_sha": "def456...", "last_sync_iso": "2026-05-20T15:42:00Z" }
  }
}
```

Drift detection: `git diff <last_sync_sha> -- <artifact_paths>` (working-tree against recorded SHA — includes uncommitted edits). Non-empty → drift exists. The trailing `..HEAD` form is **not** used because Vision §4's reconcile example covers uncommitted edits ("Developer edits src/auth/login.ts by hand") which `..HEAD` misses.

**Manual run.** `/lsa:reconcile` invokes the skill. Steps in §4.4.

**Session-start warning.** A SessionStart hook (declared in `lsa/hooks/hooks.json`) fires the drift check on every new Claude Code session. If any module has drift, the hook prints a one-line notice:

```
LSA: drift detected in modules [core, lsa] — run /lsa:reconcile to absorb.
```

No further action; the human chooses when to reconcile. The hook script lives at `lsa/hooks/session-start-drift-check.sh` and consumes `.lsa.yaml` + `.lsa-sync-state.json`. The manifest schema (per `code.claude.com/docs/en/hooks`) is:

```json
{
  "description": "LSA — surface artifact drift at session start",
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/hooks/session-start-drift-check.sh",
            "timeout": 10
          }
        ]
      }
    ]
  }
}
```

`matcher: "startup"` (vs `resume`/`clear`/`compact`) is the right granularity — drift state doesn't change between session resume events. Single-file `hooks.json` is the Claude Code plugin convention; not a `session-start.json`.

**Risk: `.lsa-sync-state.json` and git history rewrites.** If the recorded SHA disappears (force-push to main, rebase), the diff command errors. Fallback: skill treats the error as "first sync" — proceeds with HEAD as the new baseline, prints a warning. Documented in the skill body's Constraints.

## 8. Doc-mode verify

**Code-mode (current, v0.1.1):** `lsa-verify` runs `git diff main -- specs/features/<name>/` vs `git diff main -- src/`, checks every `/src/` change traces to a spec requirement (`lsa/ARCHITECTURE.md:253`).

**Doc-mode (new, v0.2.0):** `lsa-verify` reads `.lsa.yaml`. For each changed file under any module's `artifact_paths`, the change must correspond to a requirement in that module's `spec`. Specifically:

1. Compute `git diff main -- vision/specs/features/<name>/` — list new/changed ACs.
2. For each module in `.lsa.yaml`, compute `git diff main -- <artifact_paths>`.
3. For each changed file in (2), confirm it's covered by an AC from (1) **or** by an existing requirement in the module's `spec.md`.
4. Untraced changes → FAIL.

**"Tracing" defined.** A changed artifact is traced if **either**:
- The feature spec's `requirements.md` names the file or its containing directory in an AC, **or**
- The artifact's diff is wholly mechanical (rename, whitespace, formatting) — judged by the agent and reported as such in the verify report.

Strict trace-per-line is **not** required for v0.2.0. The bar is "is this change *intended* per the spec?" — the human reviews the verify report and confirms.

**Mixed-mode.** When `.lsa.yaml: mode: mixed`, both code-mode and doc-mode run; either failing fails the whole verify.

## 9. Metrics

Three metrics per `vision/VISION.md:157`. **Metrics fire on T3 only** — feature-scoped, written by `lsa-verify` at the end of a T3 flow on PASS. T2 has no feature spec and no sync step, so no metrics file is generated; T1 has no LSA ceremony at all. The "two-week dogfood log" (§15) covers T2/T1 measurement at a coarser grain.

For T3 PASS, `lsa-verify` writes to `${specs_root}/archive/<feature>/metrics.md`:

```markdown
# Metrics — <feature-name>

**Feature archived:** 2026-MM-DD
**Verified by:** lsa-verify

## Accuracy to the task
- ACs declared: <N>
- ACs satisfied: <M>
- **Score:** M/N

## Proven facts with sources
- Factual claims in feature spec: <N>
- Claims with valid source + searchable quote: <M>
- **Score:** M/N

## Only-required-changes
- Files in artifact_paths changed: <N>
- Files covered by an AC or spec requirement: <M>
- **Score:** M/N (1.00 = no scope creep)
```

`lsa-sync` aggregates these on each merge into a repo-level `/specs/metrics.md` (one row per archived feature, append-only). The aggregate is human-readable but optional — the per-feature `metrics.md` is the source of truth.

**Statistical rigor explicitly deferred** per Vision §6 adjust #3 (`vision/VISION.md:222`) — pass/fail counts only, no Wilson CIs.

| Session-start hook | Open a new session after a manual SKILL.md edit → expect drift notice in stdout. |

### V3 — behavior change is observable

Two separate sub-probes — they exercise different code paths and shouldn't be conflated in a single linear flow (the verify skill expects an active feature; the reconcile skill expects drift without a feature).

**V3a — T2 feature-flow probe (exercises tier-selector + discover + doc-mode verify).**
On a feature branch in this repo:
- Pick a small T2-sized change targeting an **artifact** (file in some module's `artifact_paths`), not a spec file. Example: *"In `core/skills/ground-rules/SKILL.md`, rephrase one sentence in the 'No fake confidence' section to remove an ambiguity."* This is class (a) drift — the spec already describes 'no fake confidence' behavior; the change is wording.
- `core/tier-selector` proposes T2; human confirms.
- `lsa-discover` runs the three-question probe; agent answers (module: `core`; change: clarify wording; AC: meaning preserved).
- Agent makes the edit on the artifact.
- `lsa-verify` in doc-mode confirms: the changed file is in `core.artifact_paths`; the diff is mechanical/wording (per design §8's "wholly mechanical" trace clause); PASS WITH WARNINGS or PASS.
- **No `metrics.md` is written for T2** — metrics fire only at lsa-sync time, and T2 has no sync step (per design §9 clarification). T2 verification ends at the trace report.

**V3b — reconcile + session-start drift probe (exercises lsa-reconcile + the SessionStart hook).**
Separately, and **only after Phase C Task 30 has written `vision/specs/modules/core/spec.md`** (the reconcile skill writes back to that file):
- Make a non-breaking edit to a SKILL.md body paragraph (NOT frontmatter — frontmatter errors break the skill load before the session-start hook can run): change one word in `core/skills/ground-rules/SKILL.md`'s "no fake confidence" example paragraph. Don't commit yet.
- Close and reopen the Claude Code session → expect the SessionStart hook to print `LSA: drift detected in modules [core] ...`.
- Invoke `/lsa:reconcile` → expect per-module diff summary, classification (class a — wording change to existing requirement), and a hard confirm.
- Confirm → `core` module spec edited in-place (class a); `.lsa-sync-state.json` updated.

Eyeball both for v0.2.0. Instrument later (Vision §6 deferred).

## 15. Open follow-ups (post-0.2.0)

- **`core/registry` resurrection.** If a second pack arrives and starts duplicating the lazy-load logic Claude Code's native discovery doesn't cover, design the skill in core v0.3.0.
- **Tier-selector chain-of-thought training.** Vision §4 (`vision/VISION.md:133`) notes that human corrections to tier calls become few-shot training data. v0.2.0 ships with the four worked examples from Vision §4; a `core/tier-selector/EXAMPLES.md` companion file may grow over time.
- **`lsa-discover` → `lsa-specify` handoff format.** The scratch `discovery.md` is a working contract. If T3 invocations frequently want richer handoff (architecture sketch, dep graph), formalize it in v0.3.0.
- **Doc-mode strict tracing.** v0.2.0 accepts "intended per spec" as the trace bar. If untraced changes become a real problem in dogfood use, tighten to per-line tracing in a later version.
- **Marketplace dependency field.** Adopt when Claude Code's plugin manifest supports it.
- **Two-week dogfood log.** Capture every tier call, every reconcile run, every verify outcome on this repo for the first two weeks. Use the log to validate the ~90% trigger thresholds and inform v0.3.0 priorities.
- **Retro habit format (v0.3.0).** Per Vision §5 (`vision/VISION.md:159`): a `/specs/retro.md` (or equivalent) scratchpad of mistakes and fixes, with a promotion path into `standards/` or new skills. v0.2.0 defers; v0.3.0 designs both file format and the promotion gate.
- **Self-eval harness (v0.3.0).** Per Vision §5 (`vision/VISION.md:160`): structural checks (every actor has its sections), boundary checks (no Knowledge file holds execution flow), banned-hedge-word lint. Implementable as a `core` skill once the surface stabilizes.
- **T2 `metrics.md` location (resolved as: no metrics for T2 in v0.2.0).** v0.2.0 emits metrics for T3 only — see §9. If T2 use becomes the dominant flow, design a coarser-grain metrics surface (per-day or per-week aggregate from the two-week dogfood log) in v0.3.0 rather than per-T2-task files.
- **Tier-selector as always-on (Vision §3 amendment).** Vision §3 (`vision/VISION.md:106`) places `tier-selector` as on-demand only. v0.2.0 extends this by adding a `CLAUDE.md` fragment in `core/CLAUDE.md` that mandates invoking `tier-selector` before any non-trivial task — making the *invocation rule* always-on while the *skill body* stays on-demand. Recorded as an amendment to Vision §3. Justification: orchestration that must fire on every task can't rely on description-match alone (Round 3 Q2 dialog). Revisit if a future Vision revision incorporates this directly.
- **Reconcile classification automation.** Class (a)/(b) classification (§4.4 Step 3) is currently agent-judged. If misclassifications become a real problem (e.g., agent appends when it should update in-place), design a more deterministic check — e.g., per-requirement IDs the reconcile can match against.
